Playbook

Chris's Self-Serve Runbook for Phases 5–7

Phase 5 — Hub Network Integration

5.1 — Prepare the Transit VLAN

  1. Open Meter Dashboard → VLANs. Confirm what VLANs exist from auto-config.
  2. Look for the transit VLAN. If Sudarsshan/Alex pre-configured it, it's already there. If not, note what ID they want — don't create it yourself.
  3. Ask Parker or Jesse: "Which port on your existing switch will we use for the uplink to Meter?" Write down: port number, switch name, current VLAN assignment.

5.2 — Make the Physical Connection

  1. Patch cable from agreed port on existing DC switch → designated port on Meter core switch (likely MDF-C01 or MDF-C02 — check topology).
  2. Confirm port comes up with link on both sides. Dashboard → switch port status → should show connected.
  3. Verify port config on Meter side (trunk or access on transit VLAN). If not configured, tag Sudarsshan rather than changing it yourself.

5.3 — Verify Downstream Servers

  1. Get server list from Parker/Jesse: server name, IP, VLAN, function.
  2. From a device on the Meter network, ping each server IP. Record results:
Server: __________ | IP: __________ | Ping: ☐ Pass ☐ Fail
Server: __________ | IP: __________ | Ping: ☐ Pass ☐ Fail
Server: __________ | IP: __________ | Ping: ☐ Pass ☐ Fail
  1. If pings fail, check these three things before escalating:
    (a) Is the transit VLAN port actually up on both sides?
    (b) Is there a route on the Meter firewall for the server subnet?
    (c) Is there a route on the existing DC side pointing back to the Meter transit IP?
    If all three look right and it still fails → escalate to Sudarsshan.

5.4 — Document Site-to-Site Readiness for Clyde

  1. Write down the Meter hub-side subnet (transit VLAN subnet or server VLAN).
  2. Note for Sudarsshan what the return route from existing network to the spoke will look like. Drop it in #deployments thread.
🚨 ESCALATE IF:
  • Transit VLAN doesn't exist in Dashboard
  • Pings fail after checking items (a)(b)(c)
  • Parker/Jesse can't identify server IPs
Phase 6 — HA & PDU Validation

6.1 — Confirm HA Status

  1. Dashboard → Security Appliances. Confirm both Primary and Backup SA listed.
  2. Check HA status: Primary = "Active", Backup = "Standby"/"Passive".
  3. If both show Active, or both show Standby → screenshot and tag Sudarsshan immediately. Do not proceed.
  4. If only one SA appears → check power and cabling. If powered and cabled correctly but not showing → escalate.

6.2 — Label PDU Outlets

  1. At the rack, compare each PDU outlet to the WO power outlet diagram.
  2. Every outlet labeled with device it powers (e.g., "Outlet 1 — Primary SA").
  3. If labels missing → have Donny label now. Photo for CompanyCam.

6.3 — Run the Failover Test

  1. Open Dashboard → Security Appliances page (watch status in real time).
  2. On PDU #1, power off the outlet for the Primary SA. Start timer.
  3. Watch Dashboard → Backup SA should transition to Active within ~15 seconds.
    • If no flip within 30 seconds → power Primary back on → escalate.
  4. With Backup Active, ping downstream servers from Phase 5. Connectivity should hold.
  5. Leave Primary off ~2 minutes, then power back on.
  6. Watch Dashboard → Primary comes back online, resumes Active role (2–3 min).
  7. Ping again to confirm stability after failback.

6.4 — If HA Is Limited (Single Static IP)

  1. Still run failover test on the working WAN link.
  2. Document: "WAN1 not configured for HA — single static IP only. Failover tested on WAN0 only."
  3. Post in #deployments as deviation.
🚨 ESCALATE IF:
  • Both SAs show same HA state
  • Backup doesn't take over within 30s
  • Primary doesn't resume Active after reboot
  • Connectivity drops during failover
Phase 7 — Speed & Connectivity Tests

7.1 — Dashboard Speed Test

  1. Dashboard → ISP/WAN page → Run speed test.
  2. Compare to ISP circuit specs (check WO ISP Details section or ask Sudarsshan).
  3. Results should be ~80–90% of rated speed.
    • If getting <50% of rated → check SFP type, cable, port speed negotiation before escalating.

7.2 — Dashboard Ping Test

  1. Run ping test. Need 100% success rate.
  2. If packet loss → run again (rule out transient).
  3. If persistent loss → check WAN quality in Dashboard.
    • WAN quality degraded → ISP-side issue
    • WAN quality fine but ping fails → escalate

7.3 — Client Speed Test

  1. Connect laptop to Meter network via Ethernet.
  2. Browser → speedtest.meter.network → Run test.
  3. Screenshot results (confirm showing wired connection, not LTE).
  4. Upload screenshot to CompanyCam.

7.4 — N/A Items

  1. Skip and note as N/A: Guest WiFi, Wireless Health, ACS (no WAPs at this site).
🚨 ESCALATE IF:
  • Speed <80% of rated after checking SFP/cable/port
  • Persistent ping packet loss
  • WAN quality below 100% in Dashboard
